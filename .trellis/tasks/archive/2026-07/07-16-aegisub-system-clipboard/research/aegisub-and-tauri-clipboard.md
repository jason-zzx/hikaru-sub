# Aegisub and Tauri clipboard research

## Aegisub behavior

Source snapshot: TypesettingTools/Aegisub commit `4dfc7b2e5d5c861e337050a4fe8b708c6525f355`.

- Row copy serializes each selected `AssDialogue` with `GetEntryData()` and joins rows with CRLF. This is a complete `Dialogue:`/`Comment:` event line, not text-only content. Evidence: [`src/command/edit.cpp` lines 553-557](https://github.com/TypesettingTools/Aegisub/blob/4dfc7b2e5d5c861e337050a4fe8b708c6525f355/src/command/edit.cpp#L553-L557).
- Outside a focused text control, paste splits clipboard text on CR/LF and handles each non-empty line independently. Each line is parsed as ASS; a parse failure becomes a new default dialogue whose `Text` is that line. All created rows are inserted immediately before the active row. Evidence: [`src/command/edit.cpp` lines 99-138](https://github.com/TypesettingTools/Aegisub/blob/4dfc7b2e5d5c861e337050a4fe8b708c6525f355/src/command/edit.cpp#L99-L138) and [`src/command/edit.cpp` lines 850-880](https://github.com/TypesettingTools/Aegisub/blob/4dfc7b2e5d5c861e337050a4fe8b708c6525f355/src/command/edit.cpp#L850-L880).
- Therefore mixed valid/invalid clipboard lines are accepted line-by-line outside text controls: valid lines retain ASS fields; invalid lines become text-only default rows.
- Aegisub's plain-text fallback uses layer `0`, start `0`, end `0` (the fallback explicitly overrides the default 5-second end), style `Default`, and the trimmed input line as text. Evidence: [`src/command/edit.cpp` lines 99-113](https://github.com/TypesettingTools/Aegisub/blob/4dfc7b2e5d5c861e337050a4fe8b708c6525f355/src/command/edit.cpp#L99-L113) and [`src/ass_dialogue.h` lines 135-148](https://github.com/TypesettingTools/Aegisub/blob/4dfc7b2e5d5c861e337050a4fe8b708c6525f355/src/ass_dialogue.h#L135-L148).
- With a text control focused, Aegisub first attempts an all-or-nothing ASS-line paste only when the trimmed clipboard begins with `Dialogue:`; if that fails it delegates to the focused control's native paste. Hikaru Sub already prevents whole-row hotkeys from matching `input`, `textarea`, and `contentEditable`, so its existing native text behavior should remain untouched. Evidence: [`src/command/edit.cpp` lines 819-880](https://github.com/TypesettingTools/Aegisub/blob/4dfc7b2e5d5c861e337050a4fe8b708c6525f355/src/command/edit.cpp#L819-L880).
- Paste is enabled only when the clipboard advertises text/unicode text; a non-text clipboard is not processed. Evidence: [`src/command/edit.cpp` lines 857-865](https://github.com/TypesettingTools/Aegisub/blob/4dfc7b2e5d5c861e337050a4fe8b708c6525f355/src/command/edit.cpp#L857-L865) and [`src/utils.cpp` lines 134-146](https://github.com/TypesettingTools/Aegisub/blob/4dfc7b2e5d5c861e337050a4fe8b708c6525f355/src/utils.cpp#L134-L146).

## Tauri 2 clipboard manager

Source snapshot: tauri-apps/plugins-workspace `v2` commit `40ae0a7fa0ecc9b0cdbd952297f7a17f37308229`.

- Add `tauri-plugin-clipboard-manager = "2"` to `src-tauri/Cargo.toml`, `@tauri-apps/plugin-clipboard-manager` to `package.json`, and register `.plugin(tauri_plugin_clipboard_manager::init())` on the Tauri builder. Evidence: [plugin README](https://github.com/tauri-apps/plugins-workspace/blob/40ae0a7fa0ecc9b0cdbd952297f7a17f37308229/plugins/clipboard-manager/README.md#L27-L70).
- `writeText(text)` returns `Promise<void>` and `readText()` returns `Promise<string>`. Evidence: [`guest-js/index.ts` lines 13-48](https://github.com/tauri-apps/plugins-workspace/blob/40ae0a7fa0ecc9b0cdbd952297f7a17f37308229/plugins/clipboard-manager/guest-js/index.ts#L13-L48).
- Clipboard access is disabled by default. The main-window capability must explicitly include `clipboard-manager:allow-read-text` and `clipboard-manager:allow-write-text`. Evidence: [`permissions/default.toml`](https://github.com/tauri-apps/plugins-workspace/blob/40ae0a7fa0ecc9b0cdbd952297f7a17f37308229/plugins/clipboard-manager/permissions/default.toml) and [permission reference](https://github.com/tauri-apps/plugins-workspace/blob/40ae0a7fa0ecc9b0cdbd952297f7a17f37308229/plugins/clipboard-manager/permissions/autogenerated/reference.md#L73-L169).
- Desktop `readText()` delegates to `arboard::Clipboard::get_text()` and propagates failure through the Tauri command. A clipboard without readable text can therefore be implemented as a caught read failure/no-op, satisfying the non-text requirement without requesting image access. Evidence: [`src/desktop.rs` lines 62-69](https://github.com/tauri-apps/plugins-workspace/blob/40ae0a7fa0ecc9b0cdbd952297f7a17f37308229/plugins/clipboard-manager/src/desktop.rs#L62-L69) and [`src/commands.rs` lines 35-40](https://github.com/tauri-apps/plugins-workspace/blob/40ae0a7fa0ecc9b0cdbd952297f7a17f37308229/plugins/clipboard-manager/src/commands.rs#L35-L40).

## Repository mapping and planning implications

- Current hotkey and context-menu paths duplicate clipboard orchestration and share a module-level `SubtitleCue[]` clipboard in `src/services/editorActions.ts`; system clipboard operations are asynchronous, so both entry points should call one shared async subtitle clipboard service.
- The current `parseAss()` only recognizes events inside an `[Events]` section and its time/number parsing is intentionally forgiving. Clipboard parsing needs an explicit event-line validator/parser rather than using `parseAss()` output length alone as validity proof.
- Existing `pasteCueRows()` inserts after the target, while Aegisub inserts before the active row. The user explicitly preserves the old effect for valid ASS paste but explicitly requests before-selection insertion for non-ASS text, so these two insertion rules should remain distinct.
- The context menu cannot synchronously know whether the external system clipboard contains text. Its Paste item should remain enabled when a target exists and let the async read decide whether the action is a no-op.
- `replaceCues()` creates one undo-history entry, so each successful multi-row paste/cut should continue to apply one consolidated cue-list result.
